# SlintUI 项目协作与生产级组件 API 规范

本文是本仓库的项目级 Agent 工作指令，也是 SlintUI 后续组件设计、实现、评审和 Gallery 展示必须遵循的生产级 API 规范。任何组件都必须先形成可使用的公开契约，再进入视觉实现；只有界面、只有导出、只有静态演示，均不视为组件完成。

现有组件默认按 `Alpha` 重新审视。历史文档中的“已实现”只表示仓库中存在对应源码和演示入口，不自动表示 API 完整、行为稳定或达到生产标准。

## 1. 基本原则

- 每个组件必须分别定义“公开 API、视觉规范、行为规范”三部分，不能用通用占位文字代替组件自己的完整清单。
- 三部分内部只列组件真实拥有的内容；没有数据模型、事件、槽位、公开方法或某种状态时，直接省略对应条目，不创建空章节，也不写“无”或“不适用”。
- 同一语义在不同组件中使用相同名称、类型、默认值和触发规则。
- 状态只能有一个明确所有者，不建立内部状态、外部属性和数据模型互相覆盖的多套来源。
- 用户操作、程序调用和宿主直接修改属性必须区分，禁止回调重复触发或形成绑定循环。
- 组件不内建业务文案、业务 ID、平台策略或特定产品数据。
- 组件内部不散落颜色、字号、间距、尺寸、动效时长和层级常量，统一消费 Theme Token。
- 原生能力不足时记录具体限制，采用最小必要自定义；不增加静默回退、兼容层或伪实现。
- 公开 API 变更、事件语义变更和状态所有权变更必须先讨论，再实现。

## 2. 每个组件必须分别定义的三部分

每个 `design/component-specs/<category>/<component>.md` 都必须针对该组件重新填写以下三部分。不能从其他组件复制一份通用清单，也不能为了结构整齐虚构属性、数据、事件或方法。

完整的含义是“真实公开内容没有遗漏”，不是“预设类别全部出现”。例如 Button 可以只有属性、内容、点击事件和焦点方法，不需要数据模型；Divider 可以只有属性和视觉规则，不需要事件或公开方法。

### 2.1 公开 API

公开 API 只记录调用方可以输入、绑定、监听、组合或调用的内容。根据组件实际能力，从以下类别中选择并完整列出；没有的类别直接省略。

#### 属性

每个属性必须记录：

- 公开名称和 Slint 方向：`in`、`in-out` 或 `out`。
- 类型、默认值、是否必填。
- 状态所有者以及组件是否会修改该值。
- 程序赋值、双向绑定和用户操作之间的关系。
- 合法范围、空值语义和非法值处理。
- 与其他属性的互斥、依赖和优先级。

Slint 属性方向统一如下：

| 方向 | 用途 | 示例 |
|---|---|---|
| `in property` | 调用方输入，组件不修改 | `enabled`、`variant`、`placeholder-text` |
| `in-out property` | 受控状态，可双向绑定 | `text`、`checked`、`current-index`、`open` |
| `out property` | 组件派生的只读状态 | `has-focus`、`valid`、`display-text` |

#### 数据类型与模型

只有组件实际接收结构化数据时才定义，并完整列出：

- `struct`、`enum` 及每个字段或枚举值的语义。
- 稳定标识、业务值和显示文本的区别。
- 模型所有权，以及组件能否修改模型。
- 空模型、重复标识、禁用项和当前值不存在时的处理。
- 排序、过滤或窗口数据变化后如何保持选择。
- 是否实例化全部项目，是否使用窗口数据或虚拟化协议。

禁止为没有数据集合的组件强行增加模型，也禁止用只适合 Gallery 的字符串数组冒充生产数据结构。

#### 内容与组合入口

只有组件允许调用方插入内容时才列出：

- `@children` 或其他实际可用的内容入口。
- 每个入口可放置的内容、布局区域和焦点顺序。
- 默认内容、空内容、图标与文字的组合规则。
- Slint 缺少命名 Slot 时采用的模型或宿主协议。

#### 事件

只有组件实际向调用方发出通知或请求时才定义。每个事件必须记录：

- 名称、参数类型和每个参数的含义。
- 触发来源和准确时机。
- 一次用户操作允许触发的次数。
- Disabled、ReadOnly、Loading 或 Busy 时是否触发。
- 相同值、重复操作、失败和越界路径是否触发。
- 它表示已经成功，还是只表示宿主请求。

统一使用以下语义，不为组件预先创建它不需要的事件：

| 事件 | 语义 |
|---|---|
| `clicked()` | 用户完成一次无值操作 |
| `edited(value)` | 用户正在编辑原始值 |
| `changed(value)` | 组件接受并提交一个新值 |
| `selected(index, value)` | 用户选择一个有效项目 |
| `submitted(value)` | 用户明确提交当前内容 |
| `cleared()` | 用户执行清除操作 |
| `opened()` / `closed(reason)` | 浮层实际完成打开或关闭 |
| `*-requested(...)` | 请求宿主或平台执行动作，尚不代表成功 |

#### 公开方法

只有调用方确实需要命令组件执行操作时才公开方法。每个方法必须记录：

- 方法名、参数和返回结果。
- 使用场景及不能用属性绑定替代的原因。
- 前置条件以及禁用、只读、加载、忙碌状态下的结果。
- 修改哪些 `in-out` 状态。
- 是否发出事件以及事件顺序。
- 重复调用是否幂等。
- 非法参数、空模型和边界值的结果。

`focus()`、`open()`、`close()`、`reset()`、`select()` 等只是可能的公开方法，不要求所有组件提供。

#### 全局上下文与宿主协议

只有组件实际依赖时才列出：

- Theme、Density、Locale、Direction、reduced-motion 等全局输入。
- ButtonGroup、Form、Disabled 等父级上下文。
- 文件对话框、窗口操作、系统通知等宿主或平台请求。
- 组件负责的部分与宿主负责的部分，以及成功结果如何回写。

### 2.2 视觉规范

视觉规范只描述该组件真实存在的结构、变体和状态，不复制全库状态枚举。必须根据组件实现完整列出：

- 组成结构：容器、文字、图标、前后缀、指示器、操作区、浮层等实际区域。
- Variant：每个语义变体、用途及禁止组合。
- Size 与 Shape：支持值、默认值、Density 影响和最小命中区域。
- Theme Token：组件实际消费的文字、背景、边框、焦点、功能色、间距、圆角、尺寸、层级和动效 Token。
- 状态外观：只列组件真实拥有的状态，并说明状态组合优先级。
- 内容规则：空内容、长文本、换行、截断、图标尺寸和数字格式。
- 布局规则：最小、首选、最大尺寸，拉伸、对齐、滚动、裁剪和溢出。
- 环境表现：浅色、深色、高对比度、Density、100%–200%、LTR/RTL 和 reduced-motion 中实际需要变化的部分。

视觉规则：

- Variant 表达语义，不允许调用方传任意颜色绕过 Theme。
- Size 和 Density 分离；Size 是组件参数，Density 是窗口环境。
- Hover 不承载唯一信息，Focus 不能只靠颜色。
- 属性或行为没有产生的状态，不得为了 Gallery 展示而虚构外观。
- 组件不能通过私有常量只修补某个 Gallery 示例。

### 2.3 行为规范

行为规范描述组件如何运行，不把行为伪装成属性、事件或方法。根据组件真实交互完整列出以下适用内容；没有的行为直接省略。

- 状态模型：内部状态、受控状态、只读派生状态及状态转换条件。
- 用户输入：鼠标、键盘、触摸板以及每种输入对应的操作。
- 键盘映射：Tab、方向键、Enter、Space、Escape、Home、End 等实际支持的按键。
- 事件时序：状态何时先更新、事件何时发出、宿主何时回写。
- 方法行为：公共方法与同等用户操作是否一致，以及幂等和失败规则。
- 焦点：初始焦点、焦点环、Tab 顺序、浮层焦点陷阱和关闭后的焦点恢复。
- 边界处理：空值、相同值、非法值、越界、禁用项、重复操作和失败路径。
- Loading、Busy、ReadOnly、Disabled 对输入、方法和事件的具体影响。
- 无障碍：角色、名称、描述、值、选中、展开、只读、禁用和 live region。
- 本地化与方向：文案来源、格式化、长文本、RTL 逻辑方向和方向图标。
- 动效：进入、退出、状态过渡及 reduced-motion 行为。
- 平台与资源：原生窗口、弹层、计时器、异步任务、清理和性能边界。

行为规范必须能回答“调用方做了什么、组件状态如何变化、发出什么事件、最终焦点在哪里”。纯视觉组件不需要编造点击、键盘或焦点行为。

### 2.4 单组件填写模板

每个组件复制以下结构后，只保留它真实拥有的子项：

```markdown
# ComponentName

## 公开 API

### 属性
| 名称 | 方向 | 类型 | 默认值 | 必填 | 所有者 | 说明与边界 |

### 数据类型与模型
### 内容与组合入口
### 事件
| 名称 | 参数 | 触发时机 | 阻止条件 | 重复与边界 |

### 公开方法
| 名称 | 参数/返回值 | 状态影响 | 事件 | 重复与边界 |

### 全局上下文与宿主协议

## 视觉规范

### 组成结构
### Variant、Size 与 Shape
### 状态外观
### Theme Token
### 内容、布局与环境

## 行为规范

### 状态转换
### 用户输入与键盘
### 事件时序与方法行为
### 焦点与浮层

```

例如组件没有数据模型，就删除“数据类型与模型”；没有事件，就删除“事件”；没有公开方法，就删除“公开方法”。禁止保留空标题，也禁止填写“无”“不适用”来制造表面完整度。

## 3. 公开 API 设计规范

### 3.1 命名

- 公开组件使用 `Button`、`TextField` 等语义名称，不添加 `Slint` 或 `SlintUI` 前缀。
- 属性和回调使用 kebab-case；结构体和枚举使用 PascalCase；枚举值使用小写语义名称。
- 同一语义不得同时出现 `value`、`selected-value`、`current-value` 等多个名称，除非含义确实不同并有说明。
- `request-*` 用于方法，`*-requested` 用于宿主请求回调。
- 不暴露组件内部元素名称、坐标修补属性或只服务 Gallery 的参数。

### 3.2 受控状态

每个可变状态必须选择一种模式：

1. 受控：公开 `in-out property`，组件和宿主共享同一值。
2. 请求式：组件不修改数据，只发出 `*-requested`，宿主确认后更新属性或模型。
3. 只读派生：通过 `out property` 暴露，不允许调用方写入。

禁止同一状态既由组件直接修改模型，又要求宿主通过请求回调再次修改。

### 3.3 组合槽位

- 组件支持任意内容时使用 `@children`，并定义内容区域的尺寸、padding、clip 和焦点顺序。
- 需要多个语义区域但 Slint 不支持命名 slot 时，优先采用明确模型或几何输出；必须在实现讨论中说明限制。
- 不用一组坐标属性伪装通用组合能力。
- 核心操作区必须能插入真实 Button 等组件，不能只接受操作文案字符串。

### 3.4 API 表是实现前置条件

每个组件规格必须为实际存在的公开 API 类别分别给出完整表格，不能只写一行名称。不存在的属性、数据类型、组合入口、事件或公开方法类别应直接删除，不保留空表或“无”“不适用”占位。

属性、事件和方法不得混在一张表中，因为它们的约束字段不同。以下仅为字段示例，不表示每个组件都必须拥有这些 API：

| 名称 | 类别 | 类型 | 默认值 | 必填 | 所有者 | 说明与边界 |
|---|---|---|---|---|---|---|
| `text` | 属性 | `string` / `in-out` | `""` | 否 | 调用方 | 程序赋值不触发 `edited` |

事件表必须补充触发源、时机、参数、阻止条件和单次操作触发次数；方法表必须补充前置条件、状态影响、事件顺序、幂等性和边界处理。枚举、数据结构和关闭原因必须单独列出字段和值。

## 4. 实现前必须讨论的内容

任何组件重构或新增前，先提交一份“组件实现提案”，至少包含：

1. 使用场景：至少两个真实桌面场景，以及明确的不适用场景。
2. 三部分完整清单：公开 API、视觉规范和行为规范；每部分只列该组件实际拥有的内容。
3. 状态所有权：组件实际可变状态中的 `in`、`in-out`、`out` 或请求式状态。
4. 事件时序（组件公开事件时）：正常、禁用、重复操作、失败和边界路径。
5. 原生能力：Slint 标准组件或公开 API 能复用什么，具体缺少什么。
6. 实现选择：原生包装、组合组件或最小自定义，以及选择理由。
7. 组合方式：slot、模型、宿主协议和平台边界。
8. 兼容影响：是否改变现有公开 API；当前 Alpha 默认直接修正，不保留无需求的旧接口。
9. Gallery 方案：要展示的卡片，以及组件实际需要的交互日志、方法控制和环境矩阵。
10. 测试方案：从组件实际状态、事件、键盘、焦点、边界、截图和平台约束中选择并完整覆盖。

以下情况必须暂停实现并讨论：

- 存在两种明显不同的状态所有权或数据模型方案。
- 会改变公开属性、事件参数、事件时序或方法语义。
- Slint 原生能力不足，必须选择自绘、额外窗口或宿主协议。
- 会引入平台 API、异步任务、文件系统、网络或系统权限。
- 需要牺牲键盘、焦点、无障碍、RTL 或缩放能力。

## 5. Gallery 单组件页面规则

Gallery 是组件使用文档和交互验收工具，不是静态效果拼盘。每个二级组件页面只展示一个组件，并按以下顺序组织。

页面内容必须与该组件规格逐项对应。以下条目仅在组件实际拥有相应能力时展示；没有数据模型、受控状态、事件或公开方法时，直接省略对应区域，不创建空卡片。

### 5.1 页面必须展示的信息

1. 组件名称和成熟度：Prototype、Alpha、Beta 或 Stable。
2. 一句话用途、适用场景和不适用场景。
3. 基础用法：最少参数即可工作的实例。
4. 样式：全部 Variant、Size、图标/内容槽位和布局模式。
5. 状态：全部适用的默认、Hover、Pressed、Focused、Selected、Disabled、ReadOnly、Loading、Busy 和 Validation 状态。
6. 数据（组件公开数据模型时）：真实结构模型、空值、长值、禁用项、重复 ID 或无当前值等适用边界。
7. 受控用法（组件拥有受控状态时）：属性双向绑定、宿主更新和组件更新的结果。
8. 事件（组件公开事件时）：实时事件日志，显示事件名称、参数和累计次数。
9. 方法（组件公开方法时）：使用专门按钮调用公开方法，并展示调用前后状态。
10. 键盘与焦点：列出支持按键，并提供可实际操作的焦点顺序。
11. 环境：浅色、深色、高对比度、三密度、100%–200%、中文、英文和 RTL。
12. API 表：只展示组件实际公开的属性、事件、方法、枚举、数据结构和组合入口，并完整列出各自约束。
13. Theme Token：只列真实消费的 Token，不显示通用占位文字。
14. 宿主职责与限制：平台动作、原生能力缺口、性能和尚未实现内容。
15. Slint 使用代码：展示与当前实例一致、可复制的最小代码片段。

### 5.2 示例卡片规则

- 一张卡片只说明一个主题，标题必须具体，例如“危险与加载按钮”，不使用笼统的“基础示例二”。
- 示例必须能操作，事件和状态变化可在页面内观察。
- 不把多个无关组件放进同一画布填充空间。
- 每张卡片同时提供用途说明、当前配置和预期行为。
- 禁用、只读、加载和错误状态必须真实传参，不用静态图片或说明文字代替。
- 组件拥有事件日志时，必须区分用户事件、公共方法和宿主请求。
- 代码片段必须与画布实例使用同一 API，禁止文档与实现不一致。

### 5.3 各类别最低展示集合

以下是类别级检查范围，不是要求每个组件机械展示全部条目。单组件页面只选择其规格中真实存在的能力，并把选中的每一项展示完整。

#### 操作组件

- Variant、三种 Size、文字/图标/图文、Loading、Disabled、Focused。
- 单击、重复单击、键盘激活；组件实际支持 Toggle/Checkable 时再展示其受控状态。
- `clicked` 等事件参数和次数。

#### 输入组件

- Empty、Placeholder、Populated、Focused、Disabled、ReadOnly、Required。
- Error、Warning、Success、帮助文字、清除、最大长度和长文本。
- 用户编辑、宿主赋值、提交、重置和校验的事件顺序。

#### 选择组件

- 空模型、默认选择、无选择、禁用项、长选项、当前值不存在。
- 打开/关闭、键盘遍历、选择、清除、重复选择和越界方法调用。
- 展示模型字段、业务值和显示文本的区别。

#### 布局与容器

- 最小/首选/最大尺寸、拉伸、裁剪、滚动、长内容和空内容。
- 内容槽位、标题/操作区等组合方式。
- Narrow/Standard/Wide、LTR/RTL 和 100%–200%。

#### 导航组件

- 当前项、禁用项、可关闭项、溢出、空模型和长标题。
- Tab、方向键、Home/End、Enter/Space 和焦点保持。
- 选择与关闭事件、重复请求和越界请求。

#### 数据展示组件

- Loading、Empty、Error、Populated、长文本和大数据量。
- 选择、排序、展开、重试和窗口数据等适用协议。
- 格式化值、原始值、空值和脱敏内容。

#### 反馈与浮层

- Info、Success、Warning、Error，普通与危险操作。
- Open/Close、Dismissible、Busy、重复关闭、Escape 和遮罩点击。
- 初始焦点、焦点陷阱、关闭原因、焦点恢复和层级。
- 宿主请求与实际成功事件必须分开展示。

## 6. 测试要求

交互组件至少从以下范围中覆盖自身实际拥有的路径：

- 正常操作和状态更新。
- Disabled、ReadOnly、Loading、Busy。
- 重复操作和相同值。
- 空模型、无选择、越界索引和非法参数。
- 用户操作、公共方法、宿主赋值三类路径的事件次数。
- 键盘操作、焦点顺序和关闭后的焦点恢复。
- 中文、英文、长文本、RTL 和本地化值。
- 浅色、深色、高对比度、reduced-motion、125% 和 200%。

测试必须断言组件实际公开的结果和事件，不依赖组件内部元素名称。纯视觉组件可以使用编译、语义 Token 检查和截图，并直接省略事件与方法测试。

## 7. 成熟度与完成定义

| 状态 | 定义 |
|---|---|
| Prototype | 仅用于探索，API、行为和视觉均可能变化 |
| Alpha | 三部分组件清单已写出并有基础实现，但尚未完成真实产品和平台验证 |
| Beta | API 已评审，Gallery、自动测试和至少一个真实业务页面验证完成 |
| Stable | API、行为、主题、无障碍、性能和三平台验证完成，并有版本兼容策略 |

组件只有同时满足以下条件，才能从 Alpha 升级到 Beta：

- 三部分组件清单和全部实际公开 API 表经过评审。
- Gallery 展示组件实际拥有的 API、状态、代码和限制，不建立空的数据、事件或方法区域。
- 所有适用交互路径有自动测试。
- 至少一个真实业务页面不复制组件源码即可完成接入。
- 当前主要平台完成键盘、焦点、DPI、输入法和屏幕阅读器验证。
- 已知限制不会使组件的核心用途失效。

“源码存在”“可以编译”“已从 index 导出”“Gallery 能看到”都不能单独作为完成依据。

## 8. 后续执行顺序

停止继续增加组件数量，按以下顺序重构：

1. 建立统一的组件规格结构、API 命名和 Gallery 页面模板；各组件只填写实际拥有的内容。
2. 第一批：Button、IconButton、TextField、TextArea、Checkbox、Radio、Switch、Select。
3. 第二批：Form、Tabs、PopupMenu、Dropdown、ModalDialog、ConfirmDialog、Toast。
4. 第三批：List、Table、Tree、DataGrid。
5. 用真实设置页和数据管理页验证 API，再决定其余组件的重构顺序。

每一批先完成实现提案和 API 评审，再修改代码；不得以批量复制属性和回调的方式制造表面完整度。
