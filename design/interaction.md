# SlintUI 全局交互规范

本文定义所有 SlintUI 组件共享的输入、状态、焦点、键盘、浮层、导航和反馈协议。单组件规格只能在此基础上补充特有行为，不能重新定义通用按键或关闭策略。

## 1. 交互原则

- 可预测：相同输入在相同类型组件中产生相同结果。
- 即时：按下、提交、加载、失败和完成都必须及时反馈。
- 可撤销：能安全撤销的操作优先提供撤销，而不是频繁使用确认框。
- 不丢状态：导航、调整窗口、切换主题和打开浮层不能意外清空用户输入。
- 输入等价：核心能力必须能够由键盘完成，指针和触摸板不能成为唯一入口。
- 平台一致：保留 Windows、macOS、Linux 已形成的系统快捷键与窗口习惯。

## 2. 状态模型

组件状态由四个正交维度组成：

| 维度 | 状态 | 说明 |
|---|---|---|
| 可用性 | `enabled`、`readonly`、`disabled`、`loading` | 是否允许输入以及是否正在等待结果 |
| 交互 | `default`、`hover`、`pressed`、`focused` | 当前输入设备产生的瞬时反馈 |
| 选择 | `unselected`、`selected`、`checked`、`indeterminate` | 持久选择状态 |
| 校验 | `normal`、`success`、`warning`、`error` | 输入或操作结果 |

### 2.1 状态优先级

1. `disabled` 最高：不响应 Hover、Pressed、点击、滚轮或快捷键，不触发业务回调。
2. `loading` 阻止重复触发，但保留焦点、当前尺寸和可访问名称；可取消任务必须提供独立取消入口。
3. `readonly` 可以获得焦点、选择和复制内容，但不能改变值。
4. `error`、`warning` 等校验状态与 Focus、Hover 可以并存；焦点环不得被错误边框覆盖。
5. `selected`、`checked` 是持久状态，Pressed 是瞬时状态，两者分别绘制。
6. Hover 仅在指针位于有效命中区域且组件可用时出现。

组件 API 不暴露可由运行时计算的 Hover、Pressed 状态。业务代码只控制业务状态，例如 `enabled`、`loading`、`checked` 和校验结果。

## 3. 事件触发

- Button、PopupMenuItem 等离散操作在主指针按下且于同一控件内释放时触发。
- 按下后拖出命中区域再释放不触发；重新拖回区域可恢复 Pressed。
- 键盘 Enter 或 Space 的触发时机必须与平台同类控件一致，并在组件规格中固定。
- Switch、Select、Slider 等值组件在值实际改变后触发 `changed`，程序同步同值不得重复触发。
- `clicked` 表示用户激活，`changed` 表示值变化，`submitted` 表示明确提交；禁止使用一个回调承载三种语义。
- 回调不得在渲染或主题切换时隐式触发。
- 连续输入需要节流时，由业务或数据层负责；组件保留即时视觉反馈。

## 4. 全局键盘协议

| 按键 | 全局语义 |
|---|---|
| `Tab` / `Shift+Tab` | 在组件或复合控件之间前进、后退 |
| `Enter` | 提交表单、激活默认按钮或当前可执行项 |
| `Space` | 激活聚焦按钮；切换 Checkbox、Switch、ToggleButton |
| `Arrow` | 在复合控件内部移动，或调整 Slider、NumberInput |
| `Home` / `End` | 到达复合控件首项、末项或输入首尾 |
| `PageUp` / `PageDown` | 分页、长列表或大步调整，按组件语义使用 |
| `Escape` | 关闭最上层可关闭浮层，或取消当前临时操作 |
| `Shift+F10` / Menu Key | 打开当前对象的上下文菜单 |
| `Ctrl/Cmd+A/C/X/V/Z/Y` | 遵循平台的选择、剪贴板和撤销语义 |

- 页面不得劫持文本编辑和操作系统保留快捷键。
- macOS 使用 Command 代替主要 Control 快捷键；显示文案由 `ShortcutHint` 格式化。
- 快捷键作用域按 `focused component → active overlay → active window → application` 查找，最近作用域优先。
- 快捷键冲突必须显式报错或拒绝注册，不静默覆盖。

### 4.1 复合控件

Toolbar、PopupMenu、Tabs、RadioGroup、List、Tree、Grid 等复合控件在 Tab 顺序中通常只占一个位置，内部使用方向键移动：

- 第一次进入时聚焦选中项；没有选中项时聚焦第一个可用项。
- 再次进入可以恢复上次内部焦点。
- 水平组件使用左右键，垂直组件使用上下键；RTL 下左右方向按视觉顺序处理。
- 是否允许首尾循环由组件规格说明，同类型组件保持一致。
- Tab 离开复合控件，不逐个遍历所有内部项。

## 5. 焦点管理

### 5.1 焦点可见性

- 键盘导航必须显示焦点环；纯指针点击可以只保留语义焦点，不强制显示强焦点环。
- 焦点与选中状态必须同时可辨认，不能共用完全相同的颜色和形状。
- 焦点环使用 `focus-outline`、`focus-ring-width` 和 `focus-ring-offset`，不得改变布局尺寸。
- 聚焦组件必须滚动到完整可见区域，不能被固定工具栏、状态栏或浮层遮挡。

### 5.2 初始焦点

- 普通窗口默认不强制抢焦点，沿用平台窗口激活行为。
- Dialog 打开后将焦点放在最合理的第一操作点；长内容优先聚焦标题或内容起点，避免自动滚到底部按钮。
- 危险确认框默认焦点放在安全操作，不自动聚焦危险按钮。
- 页面加载时只有在页面几乎只有一个高频主要输入时才自动聚焦。

### 5.3 焦点恢复

- Popup、Dialog 和临时页面关闭后，焦点返回触发器。
- 触发器已被删除时，焦点移动到同一列表的相邻项、所属容器或逻辑上的下一操作点。
- 删除、过滤、分页后禁止让焦点落到不存在或不可见的对象。
- 主题、密度、Locale 和布局切换必须保持当前逻辑焦点。

## 6. 指针、触摸板与拖放

- 默认主按钮执行操作；右键只打开上下文菜单，不改变业务状态，除非平台惯例要求先选中目标。
- Double Click 只用于平台已有明确习惯的场景，例如打开文件或展开树项；必须保留单击或键盘等价入口。
- Pointer 命中区域不得小于可见控件；小图标按钮通过透明内边距扩大到至少 `24 × 24px`。
- 拖放必须显示可拖动邀请、拖动预览、有效目标和禁止状态。
- 能通过拖放完成的操作必须提供菜单、按钮或键盘替代。
- 滚轮只影响指针下或当前聚焦的滚动区域；NumberInput、Select 未进入编辑状态时不得因页面滚动意外改值。

## 7. 浮层协议

### 7.1 类型与所有权

| 类型 | 是否模态 | 典型组件 | 所有者 |
|---|---|---|---|
| Tooltip | 否 | Tooltip | 被说明控件 |
| Popup | 否 | PopupMenu、Dropdown、Select、Popover | 触发器 |
| Drawer | 可选 | Drawer | 当前窗口或工作区 |
| Dialog | 是 | ModalDialog、ConfirmDialog | 当前活动窗口 |
| System | 平台决定 | 文件对话框、系统通知 | 原生窗口系统 |

- 每个浮层必须记录所有者；所有者销毁或失效时同步关闭其浮层。
- 非模态 Popup 同一所有权链默认只保持一个；子菜单属于同一链。
- 模态 Dialog 使其下层内容不可交互，并形成独立 Tab 循环。

### 7.2 关闭顺序

`Escape` 每次只关闭最上层可关闭对象：

```text
临时编辑 / 子菜单 → Popup → Drawer → Dialog
```

- Tooltip 不消费 Escape，除非它因键盘焦点显示且包含必须隐藏的内容。
- 点击外部默认关闭 Popup，不关闭包含未保存输入的 Dialog。
- 有未保存内容的关闭必须进入统一的放弃更改流程。
- 关闭动画期间立即停止接受输入；动画结束后恢复所有者焦点。

### 7.3 定位

- 浮层优先与触发器对齐，在屏幕或窗口工作区不足时依次尝试翻转、平移和限制尺寸。
- Popup 不得覆盖触发器的主要可读内容，无法避免时保留清晰空间关系。
- Tooltip 与指针保持距离，不能挡住触发目标或相邻操作。
- 多显示器环境使用当前窗口所在显示器的工作区定位。

## 8. 异步操作

- 预计小于 `300ms` 的操作通常不显示加载指示，避免闪烁。
- `300ms–2s` 使用局部 Spinner 或按钮 Loading；超过 `2s` 应提供说明，能估算时显示确定进度。
- Loading 状态阻止重复提交并保持控件宽度稳定。
- 后台任务不得用永久 Modal 阻塞主窗口；使用状态栏、通知或任务中心。
- 可取消操作必须明确取消的作用范围，并说明是否会保留部分结果。
- 失败反馈必须包含发生了什么、影响范围和可执行的下一步。
- 乐观更新仅用于可恢复且冲突风险低的操作；失败时必须回滚并说明结果。

## 9. 反馈选择

| 场景 | 首选反馈 | 约束 |
|---|---|---|
| 字段输入错误 | Inline validation | 靠近字段，提交时聚焦首个错误 |
| 页面级可恢复问题 | Alert | 保持可见，提供修复操作 |
| 简短成功结果 | Toast | 不抢焦点，可自动关闭 |
| 后台任务完成或失败 | Notification | 可跨页面保留，支持进入详情 |
| 必须立即决策 | Dialog | 仅用于阻塞性或高风险场景 |
| 长流程最终结果 | ResultState | 给出结果、原因和后续操作 |

- 同一事件只选择一个主要反馈通道，避免 Toast、Dialog 和状态栏重复播报。
- 相同消息需要去重或合并计数；错误不能被后来的成功消息立即覆盖。
- 自动关闭信息在 Hover、焦点或用户阅读时暂停计时。
- 安全、数据丢失和权限错误不得只用会自动消失的 Toast。

## 10. 导航与未保存状态

- 用户始终能够判断当前位置、上一级和可前往的位置。
- 同一目标的导航名称、页面标题和操作名称保持一致。
- 返回操作恢复合理的滚动、筛选、选择和焦点状态。
- 切换 Tab、页面、工作区或关闭窗口前，如果存在不可恢复的未保存修改，必须提示保存、放弃或取消。
- 可自动保存的设置不显示额外保存按钮，但必须提供即时状态反馈和失败恢复。
- 外部链接、文件和系统设置应明确提示将离开当前应用或打开其他程序。

## 11. 验收要求

每个交互组件在 Gallery 中至少验证：

- 指针、Tab、方向键、Enter、Space 和 Escape 的适用行为。
- 状态组合及优先级。
- 打开和关闭浮层后的焦点位置。
- 连续触发、快速切换、失败、取消和重试。
- 主题、密度、RTL、文本放大和减少动效下的交互一致性。
